# AGENTS.md

## このリポジトリでの最優先ルール

目的は、塗装会社専用AI積算・見積りシステムを、会社独自の積算ルールと既存Excel仕様を壊さずに完成させることです。実装しやすさより、ユーザーが求める実務結果を優先してください。

## 作業開始時

依頼を受けたら、最初に次を短く整理してください。

1. 今回の目的
2. 必須条件
3. 禁止事項
4. 完了判定

意図に複数の解釈があり、結果が変わる場合は作業を止めて確認してください。

## 調査段階

ユーザーが明示的に「修正OK」と言うまでは、コード、設定、依存関係、データ、ドキュメントを変更しないでください。調査では可能な範囲で次を確認してください。

1. 実際のエントリーポイント、実行コマンド、実行環境
2. 関連コードと呼び出し経路
3. Gitの変更履歴
4. 設定ファイル、入力データ、テンプレート
5. ログ、既存テスト、直近の検証結果
6. 既存の未コミット変更がある場合はその内容

推測と確認済み事実を分け、未確認事項を断定しないでください。

## 修正段階

「修正OK」後は、承認された目的と範囲内で実装、テスト、原因が同一範囲にあるエラー修正、再テストまで連続して進めて構いません。

次の場合は自動修正を止め、ユーザーへ確認してください。

- 目的、仕様、計算ルールを変える必要がある
- 既存Excelの数式、書式、セル配置を変える必要がある
- 新しい外部サービス、課金、認証情報が必要になる
- 関係のない既存機能に影響する
- データ削除、履歴改変、強制pushなど復旧困難な操作が必要になる
- 複数案で実務結果が変わり、コードだけでは判断できない

## 変更の制限

- 関係のないファイルを変更しない
- 既存のユーザー変更を上書きしない
- テンプレート内の数式セルへ値を書き込まない
- 推定値、確認値、計算値の由来を失わない
- APIキー、パスワード、顧客情報をコミットしない
- テストを通すために正しい業務要件を弱めない
- 依存パッケージ追加は、必要性と影響を説明してから行う

## 検証

変更規模に応じ、少なくとも次を確認してください。

1. 変更ファイルの構文・インポート
2. 関連する既存テスト
3. 変更した機能の正常系
4. 0、空欄、欠損、不正値などの境界条件
5. 見積書を変更する場合は、数式・書式・サンプル値残存・合計金額
6. UIを変更する場合は、実画面での操作と再実行後の状態保持

Windowsまたはブラウザ実環境でしか確認できない項目は、未確認と明記し、実行コマンドと合格条件を示してください。

## 完了報告

結論を先に書き、次だけを簡潔に報告してください。

- 変更内容
- 実行した検証と結果
- 未確認事項
- ユーザーが行う必要がある作業（ある場合のみ）

コード変更だけで完成とせず、完了判定を満たしたことを確認して終了してください。
